# Plan: Add In-Memory Cache for Whitelisted Projects

## Overview

Add an in-memory cache to `ProjectService` so that the fetched + whitelisted project list is cached for a configurable duration, avoiding repeated HTTP calls to the ArchiveTeam API on every `/metrics` scrape.

## Files to Modify

1. **`Options/ArchiveTeamOptions.cs`** — Add a `CacheDuration` property (default: 5 minutes)
2. **`Services/ProjectService.cs`** — Inject and use `IMemoryCache` in `FetchProjectsAsync`
3. **`Program.cs`** — Register `IMemoryCache` in DI

## Detailed Changes

### 1. `Options/ArchiveTeamOptions.cs`

- Add a `CacheDurationMinutes` property with a default of `5`.
- Add a computed `CacheDuration` property returning `TimeSpan.FromMinutes(CacheDurationMinutes)`.
- Support configuring via the `CACHE_DURATION_MINUTES` environment variable (handled by existing options binding).

### 2. `Services/ProjectService.cs`

- Add `IMemoryCache` dependency via constructor injection.
- In `FetchProjectsAsync`:
  - Check cache for key `"whitelisted_projects"`.
  - If cache hit → return cached `ArchiveTeamProject[]`.
  - If cache miss → fetch from API, filter by whitelist, store in cache with absolute expiration using `_options.CacheDuration`, then return.
- Use `SemaphoreSlim` to prevent concurrent cache stampedes (multiple simultaneous requests all triggering a fetch when the cache expires).

### 3. `Program.cs`

- Add `builder.Services.AddMemoryCache()` before the `ProjectService` registration.

## Caching Behavior

- **Cache key**: `"whitelisted_projects"`
- **Expiration**: Absolute expiration based on `CacheDurationMinutes` option (default 5 min)
- **Thread safety**: `SemaphoreSlim` ensures only one thread fetches when the cache is cold, others wait and reuse the result
- **Log messages**: Add log entries for cache hits and misses
